The SLIME project provides tools for JavaScript development on several platforms, including the Java platform (using the Rhino, Nashorn, or GraalVM JavaScript engines), Java servlet-based servers, the browser, and macOS's JXA JavaScript automation framework.

Getting Started

The SLIME code must be obtained, either by cloning it using git clone https://github.com/davidpcaldwell/slime.git or downloading it and unzipping it. In the future (see issue 280), it will be possible to install SLIME via a single command on a client machine.
At that point, it is ready to use; SLIME is capable of installing its own dependencies over the internet. The top-level jsh.bash script can be used to, for example, run a script that will output information describing the shell (from the source directory): ./jsh.bash jsh/test/jsh-data.jsh.js

Provided Platforms

Java

SLIME's Java support allows applications to be written in JavaScript that call out to Java platform libraries. As such, it is an ideal platform for migrating Java software toward JavaScript.
SLIME provides two Java platforms: the jsh scripting platform that runs on the JDK, and the Java servlet platform that runs on a standard Java servlet implementation. jsh also provides the ability to run a servlet implementation from within its shell.

Compatibility

JDK
Current goal is to be compatible with version used by most Java developers per most recent JetBrains survey, which is currently Java 8.
JavaScript engines
The test suite executes tests primarily with Rhino 1.7R3, with some tests for the Nashorn engine on JDK 8. GraalVM is not yet supported, although development is underway.
Servlet containers
SLIME Servlets is compatible with the Servlet 3.0.1 implementation.
jsh currently supports Tomcat 7.0.x and is tested via the test suite. Updating to Tomcat 9 is on the roadmap via issue 209.

jsh: a JavaScript application environment that can access the Java platform.

SLIME Servlets: a JavaScript environment for authoring Java servlets

Browser: a SLIME environment for the browser

Browser support is less mature than Java support, but is extensively tested in Google Chrome, Firefox and Safari.

JXA: a SLIME environment for use in macOS automation

JXA support is alpha quality.

Tools

Command-line tools

Testing

SLIME has the concept of a definition, which is a construct that provides both documentation and a test suite for a particular software module.
The best way to create definitions in SLIME is via the Fifty definition framework, which uses TypeScript to provide type definitions and documentation, and allows inline tests to be authored within those TypeScript definitions.
Some SLIME APIs are defined via the deprecated JSAPI format, which was a literate definition that allowed documentation and tests to be defined via annotated HTML.
  • Running individual definitions in Fifty:
    • tools/fifty.bash test.jsh file.fifty.ts [--part part]
    • tools/fifty.bash test.browser [--interactive] [--chrome:data pathname] [--chrome:debug:vscode] file.fifty.ts [--part part]: Invokes a script to run Fifty browser tests; see the script for command-line argument definitions.
  • (deprecated) Running individual definitions in JSAPI:
    • jsh: ./jsh.bash jsh/test/suite.jsh.js -definition definition [-part part]
    • Browser definition page: ./jsh.bash loader/browser/test/suite.jsh.js -definition pathname [...]
    • Running a JSAPI browser suite: ./jsh.bash loader/browser/test/suite.jsh.js -suite pathname [-base directory] [-interactive]

Documentation

  • tools/documentation.jsh.js: Serves the documentation for project, in a dedicated Chrome browser if Chrome is found. Typedoc documentation will be generated if necessary and served for both the top-level repository and subrepositories at local/doc/typedoc (it will be re-generated for each request if --watch is used).
    • --base directory: top-level directory of the project.
    • --host hostname: hostname to use when serving pages.
    • --index path: relative path to the index page; defaults to README.html.
    • --watch: if present, causes documentation to be re-generated for every HTML page requested; helpful when authoring docunmentation.
    • --chrome:id name: the private Chrome directory to use (under local/chrome); defaults to documentation, or document if --watch is set).

Other

  • rhino/http/servlet/tools/serve.jsh.js [--chrome:data location] [--chrome:debug:port number] [--index path-to-index] directory: Serves a directory on an ephemeral port. If --chrome:data is provided, Chrome will be opened to browse the directory, starting at the path specified by --index (default: index.html). If --chrome-debug-port is specified, the opened Chrome will allow debuggers to connect on the given port.
  • loader/api/tools/update.jsh.js [-base directory]: Inlines or updates the definition files for a library that does not itself contain SLIME (and thus cannot link to the SLIME definition files in loader/api).

wf

SLIME also provides tools for projects to implement commands related to the software lifecycle, including a top-level script that can be invoked (at tools/wf) and an associated API (in the form of a jsh plugin).
